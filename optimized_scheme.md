# 优化方案

## 架构优化

- 读写分离
  - 主节点处理写入, 从节点处理读取
  - 异步数据同步
- 代理缓存加速
  - 将root hash等重点数据放入redis进行缓存, 并不断进行更新(如每500ms/1s向数据库查询更新)
    - 客户端请求时, 可以带上自己信任的root hash
    - 代理层比对是否发生变化, 并根据不同结果进行返回
  - 缓存热点数据的merkle proof
    - 减少直接对数据库的访问请求  
- 存储对象分层
  - 大文件内容不直接存入immudb的value log
  - 将实际文件存入S3, 只将文件Hash和元数据存入immudb
  - 减少自身存储压力

## 部分安全换性能的方案

- 放宽读取一致性
  - 读取的实时性要求不是特别高时,在客户端调用 Get 或 Scan时，不强制要求 等待索引更新。允许读取“最近已索引”的数据，而不是“刚刚提交”的数据。这能极大地减少读请求对写入流程的干扰。
- 按需验证
  - 普通读取: 对于绝大多数公共访问，直接使用普通的 Get 接口（非 verifiedGet）。这省去了服务端生成证明和客户端计算哈希的巨大开销。性能提升可能是数量级的。
  - 特定验证: 仅在用户需要“审计”或“校验”数据真实性时（例如下载关键文件后），提供一个专门的“验证按钮”或接口，此时再调用 VerifiedGet。
- 不要求写后即读
  - AsyncCommit (异步提交):
    *   默认行为: 客户端发送 -> 服务端写 WAL -> 服务端更新 Merkle Tree -> 服务端落盘 -> 返回 ACK。
    *   优化: 允许服务端在写入 WAL 后立即返回 ACK，Merkle Tree 的更新和索引构建在后台异步进行。
  

## 底层源码优化方案

- 保证事务提交时使用了全局锁, 使得事务提交, AHTree追加和TxLog保证串行提交, 导致效率低下

  - 多个协程需要commit的时候,  进行组提交, 将多组协程的txheader写入txlog. 

- 客户端在进行任何读取操作（如 Get ）前，通常需要获取当前数据库的最新状态（ CurrentState ）。目前这通常涉及对 commitStateRWMutex 的 RLock 。在高频读取的公共平台上，这种频繁的读锁申请会产生不必要的 CPU 开销。

  - 使用 Atomic Value 替换读写锁保护状态。
  - 实现思路 ：

  1. 在 ImmuStore 中使用 atomic.Value 来存储最新的 TxHeader 镜像。
  2. 写事务在提交成功后，通过原子操作更新该镜像。
  3. 读取操作直接通过 atomic.Load 获取最新状态，完全不经过互斥锁。

  - 预期效果 ：实现读取操作的“零锁化”，消除公共读取压力对数据库核心状态锁的竞争。

- SHA256的指令集优化

  - 目前的SHA256是使用go语言的官方库, 如果需要极致的性能提升, 可以使用一些SIMD指令集的哈希库或AVX-512指令集去做针对性优化.来加速生成merkle tree的过程.

- 目前的 History 检索会将历史版本的 Value 随同 TxHeader 一起加载。对于公开数据平台，用户可能只想看历史列表的相关数据（如修改时间、版本号），并不一定需要立即读取所有旧版本的大块数据。

  - 修改 History 接口，支持“仅元数据”模式。
  - 在回溯 hLog 时，只读取事务元数据，跳过 VLog 中的具体值。
